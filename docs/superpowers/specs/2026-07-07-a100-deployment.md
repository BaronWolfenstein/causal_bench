# A100 Deployment Spec — causal_bench GPU work

**Status:** living document. Covers what's built today (SMC/IPCW core, localization diagnostic). The **diffuse_directly** section is a stub to fill when that work lands (§6). QLoRA/LLM fine-tuning is **not** causal_bench — it lives in SGA (Scier/Mount Sinai portfolio).

**Target hardware:** on-prem **8× A100 40GB** Lambda box, reached over a Tailscale tunnel (not metered cloud — constraints are physical/operational, not billing). See the localization GPU build spec for the encoder-side detail; this spec is the deployment/porting view across the whole repo.

**Core principle carried from all prior design:** stay at the **collective / library layer**. The multi-GPU barrier is an NCCL collective; NCCL owns device-side sync, stream ordering, and memory fencing. Do **not** hand-write CUDA kernels, cooperative-groups grid syncs, or CPU memory fences (membarrier) — every one of those is below the boundary and means reimplementing NCCL/CUB. Systematic resampling is `cupy.cumsum` + `cupy.searchsorted` (CUB `DeviceScan`), already optimal and self-synchronizing.

---

## 0. Environment & dependencies

- **Torch (CUDA 12 build)** + **`cupy-cuda12x`** — add under a `[gpu]` extra in `pyproject.toml`; never import at module top level (lazy-import inside functions so CPU-only installs and CI stay torch-free). The sampling package already lazy-imports cupy in `backend.array_namespace`.
- **Localization encoder stack:** `femr`, `transformers`, `torch` — **version-coupled**; pin to the MOTOR/CLMBR model-card README exactly at load time.
- **Device resolution:** `resolve_device("auto")` (cuda → mps → cpu) for single-device torch; `cuda_available_devices()` (reads `CUDA_VISIBLE_DEVICES`, falls back to `torch.cuda.device_count()`, returns `[]` on a CPU box) gates the multi-GPU path.
- **Shared-box discipline** (the box is likely multi-user — "8×A100" may be "3 free right now"):
  - `nvidia-smi` before launching; note idle GPUs.
  - Pin with `CUDA_VISIBLE_DEVICES=<free, NVLink-adjacent ids>` **before** `torchrun`; `--nproc_per_node` takes the first N *visible* devices.
  - `nvidia-smi topo -m` to confirm NVLink adjacency (`NV#` = NVLink; `PHB`/`PXB`/`SYS` = PCIe). On an NVSwitch box every pair should read NVLink.
  - Every long job inside `tmux`/`screen` — a Tailscale reconnect or closed lid must not kill a multi-hour run.
  - **Tailscale is the control plane, not the data path** — land/generate data on the box's local disk; do not push large MEDS/embedding parquet across the tunnel.
- **Storage:** run `df -h` first. Footprint is small (embeddings parquet tens of MB–GB; particle state is MB; score-net checkpoints small). Internal NVMe is the hot-read path. An external TB5 SSD is **not needed** at these sizes — only helps cold snapshots/dataset-at-rest if the box is genuinely out of space, and even then keep hot reads on internal NVMe.

## 1. SMC/IPCW core → multi-GPU (the deferred sub-plan, made concrete)

Today: `causal_bench/sampling/` is numpy-first; `run_smc(device="cpu")` is the validated reference. `device="cuda"` is a **scaffolded seam only** — the hot loop still calls `np.*`. Deployment work, in order:

### 1a. Port the hot loop to the array namespace `xp`
The one real code gap flagged in review. Route through `xp = array_namespace(device)`:
- `smc_step`: the resample branch's `np.zeros(len(idx))` → `xp.zeros`; `state.particles[idx]` gather works on cupy.
- `weights.normalize_log_weights`: `np.max`/`np.exp`/`np.log` → `xp.*` (keep the all-`-inf` raise — it now surfaces total weight collapse loudly).
- `resample.systematic_resample`: `np.cumsum`/`np.searchsorted` → `xp.*` (cupy backs these with CUB — do not reimplement).
- Keep `run_smc` returning **host numpy** via `to_numpy(...)` (already done) so callers/estimators stay device-agnostic.
- Acceptance: `run_smc(device="cuda")` numerically matches `run_smc(device="cpu")` on the same seed to tolerance.

### 1b. Collectives — the barrier
- **`all_reduce(Σw, Σw²)`** across ranks → Kish ESS (the only true sync point). Fuse the **weight-finiteness check** into this barrier (cheap; catches the all-`-inf` collapse at the reduce).
- **`all_gather`** weights so every rank holds `w[1..N]`.
- **`all_to_all`** particle redistribution (a survivor on rank 3 may be ancestor of particles owned by rank 0).
- Via `torch.distributed` (NCCL). No custom kernels.

### 1c. Gating & topology
- `cuda_available_devices()` → if empty, **fall back to single-process CPU `run_smc`** (never crash on a CPU box). If non-empty, init the process group over exactly those ids, `nproc_per_node = len(ids)`.
- Pin NVLink-adjacent ids; set `CUDA_VISIBLE_DEVICES` before `torchrun`.

### 1d. Minimize data movement (roofline: the gather is memory-BW-bound, ≈0 arithmetic intensity)
- **Island / local resampling** as the default: each rank resamples its own sub-population, occasional global exchange only — removes the `all_to_all` at a small bias/variance cost.
- **Ancestor-index indirection:** keep the int32 lineage vector (already produced), defer the physical state gather until states are next written — avoids redundant copies when one survivor is duplicated massively (the rare-event degeneracy regime).

### 1e. Correctness & validation on-box
- **Shared-seed systematic resampling → byte-identical indices across ranks.** The numpy oracle (`sharded.sharded_systematic_resample`, Task 6) is what the real `all_to_all` must match.
- **Decisive test:** distributed run == single-GPU reference SMC on the same seed, to numerical tolerance. Validate at **2 ranks** first (enough to exercise `all_reduce` + `all_to_all`).
- Then the **throughput sweep** at 2/4/8 ranks (free on your own box; was previously out of scope).
- Post-resample weights normalize to uniform on every rank (assert).

## 2. Localization diagnostic → real encoders (deployment view)

The decision procedure (`run_diagnostic`, Tests A/B/B′/B″/C) is **encoder-agnostic numpy — it runs unchanged on any machine.** Only the *production of its input arrays* is GPU work:

- **Frozen encoder:** MOTOR (`StanfordShahLab/motor-t-base`, primary) or CLMBR-T-base (fallback) via FEMR; `eval()`, `torch.no_grad()`, no gradients; pinned FEMR/transformers/torch versions. Research stand-in for SMB's JEPA — non-commercial license, diagnostic only, do not ship.
- **MEDS cohort:** extract-once → persist embeddings to parquet; all downstream iteration is GPU-free.
- **`E_eval` decoupled encoder (#88):** a **second** frozen encoder (e.g. MOTOR generates, CLMBR evaluates). The full Tier-2 metric-hacking guard needs generated samples rendered to MEDS then re-encoded by `E_eval` — that render step is ELF discretization and lands with **diffuse_directly** (§6). Tier-1 (separation AUC of the real reference sets in `E_eval`) is available immediately.
- **Recon / guided arrays** (`recon_b`, `rare_guided`, …) are produced by the diffusion (§6) and fed to `run_diagnostic`, which then returns the terminal.

## 3. Profiling on-box — what only the box can tell us

**Measure on the A100 (uninformative off-box):**
- Absolute wall-clock; **all-to-all comm cost** (O(N·dim) over NVLink); memory-bandwidth / batch-throughput tradeoffs.

**Already CPU-validated — do NOT re-derive on-box:**
- Resample-trigger rate (property of how far R sits from base mass), per-particle scaling (O(N), not O(N²)), ESS/weight-degeneracy numerical health, and the distributed==serial index invariant. These are hardware-independent and settled.

**Roofline split to hold in mind:** score net (§6) → *maximize FLOPs* (bf16/Tensor Cores, large batch, `torch.compile`); resampling gather → *minimize movement* (island / index-indirection). Opposite sides of the roofline.

---

## 4. Checkpointing

- **SMC has no trainable weights** → no weight/optimizer checkpoints. Only persist embeddings/particle state to parquet for long runs / resume.
- **Fault tolerance:** for any long GPU job, snapshot enough to resume after crash/preemption (shared box). For SMC that's the particle ensemble + RNG state + step index.
- (Training checkpoints — weights + Adam state, resume/rollback — belong to **diffuse_directly** §6. Note: *activation/"gradient" checkpointing* is a VRAM technique, not disk. Checkpoints are needed even when the model fits in VRAM — for crash recovery and model selection, not memory. At our sizes they're small; internal NVMe suffices, TB5 SSD unnecessary.)

---

## 5. Honest boundaries / non-goals

- **No custom CUDA kernels, cooperative groups, grid/warp/block sync, or CPU memory fences (membarrier).** Collective layer only; cupy/CUB for scan/searchsorted.
- **`device="cuda"` for SMC is not yet end-to-end validated** — §1a is the prerequisite before any cuda claim.
- The localization decision procedure is not GPU work and must not be "ported" — it stays CPU/numpy.
- **The zero-flow CI estimator (`detectors/zero_flow_ci.py`) stays CPU too.** It's numpy/sklearn — like the localization decision procedure, it needs no port. Only its optional torch neural-velocity backend (§7) is GPU work, and only in the high-dim × large-n regime.

## 6. diffuse_directly deployment

The **CPU machinery is built** (plan: `2026-07-07-diffuse-directly.md`; T1–T9 on main — numpy core, torch score net T8, ELF render/#88 T9, training checkpoints). This section is the **deployment / on-box view**: what remains is the perf layer (bf16/compile), the real encoder, and the on-box CUDA validation (issue #116) — not the machinery itself. **QLoRA/LLM fine-tuning is out of scope for causal_bench** — it lives in the scientific-graph-agent repo (Scier/Mount Sinai portfolio, `2026-07-07-sga-portfolio-step2.md` §C); do not fold it back into this spec.
- **Torch score net:** `resolve_device`; train on ZCA-whitened embeddings; tail-aware weighted loss. **BUILT (CPU, PR #96, `generative/score_net.py`)** — device-agnostic, auto-runs on cuda when detected. **The bf16 autocast (Tensor Cores) + large-batch + `torch.compile` "maximize-FLOPs" perf layer is NOT built** — the net runs **fp32** on cuda until it lands (CUDA-only, validatable only on-box). On-box `cuda==cpu` parity smoke test: **issue #116**.
  - **Manifold-aware tangent-DSM variant (causal_bench #108, gated):** the tangent-space-penalty score training (`generative/tangent_dsm.py` — the CPU closed-form prototype merged in PR #106, with learned-metric normals via local PCA) is a *training-time* variant of this same score net — identical GPU path (bf16/Tensor Cores, `torch.compile`, `resolve_device`, ZCA-whitened embeddings) plus a differentiable tangent penalty as an auxiliary loss (SGD, so the RFF/ridge closed form is lost). The CPU closed-form stays the **validation oracle**; the torch net is the high-dim consumer. `estimate_local_normals` (k-NN + local-PCA SVD) is CPU preprocessing and does **not** gate the GPU loop. Adopt only if a curvature / STRUCT-S detector fires (manifold-aware propensity spec, PR #99). Same real-embedding data gate as the rest of §6.
- **ELF final-step discretization + `E_eval` render→re-encode** — completes the Tier-2 #88 guard on real encoders.
- **Two-space eval:** one space to generate-and-match (ELF), a decoupled `E_eval` only to score fidelity; learned latent only if the diagnostic returns `separate_latent_justified` (architecture designed in `2026-07-11-separate-latent-design.md`).
- **Twisted-SMC at inference:** the §1 multi-GPU SMC is the inference-time reranker for the `smc_required` terminal — CFG at training, SMC at inference. **The multi-GPU/NCCL SMC port is its own on-box track (§1a-1d, PR #94)** — separate from the single-device score net here.
- **Training checkpoints:** resume/rollback (score-net weights + optimizer state; small at our sizes). **BUILT (CPU, PR #111, `generative/checkpoint.py`).** Cross-device resume is handled: a checkpoint saved on cpu (`map_location='cpu'`) resumes on cuda because `train_score` re-aligns the Adam moments to the model's device before stepping. On-box confirmation (resume + rollback on cuda) is in the **issue #116** checklist.
- **Real MOTOR/CLMBR pipeline:** swap the stand-in `RandomProjectionEncoder` for the frozen EHR encoder behind the same `FrozenEncoder` signature.

## 7. Deferred: zero-flow CI — torch neural-velocity backend (spec exists)

The zero-flow CI estimator (`detectors/zero_flow_ci.py`) is CPU numpy/sklearn and needs **no** porting. Its **only** GPU-relevant piece is the optional neural-velocity backend, fully specced in `docs/superpowers/specs/2026-07-08-zero-flow-torch-extension.md`. Deployment implications when/if built:

- **Warranted only for high-dim × large-n CI** (e.g. independence between learned/patient embeddings). Tabular CI stays CPU sklearn; moderate nonlinearity uses an sklearn RF/GB velocity — neither touches the box.
- **`[flow]` extra** (`torch`), lazy-imported (same discipline as `[gpu]`); `resolve_device`; bf16 autocast on CUDA. `velocity_factory=TorchVelocity` drops in with no API change; `predict` returns host numpy so the test/permutation logic stays device-agnostic.
- **Cost driver = the permutation calibration**, not a single fit: `n_perm × cross-fit` velocity fits. The GPU lever is **batching the per-permutation velocity fits** (and the per-fold ones), not accelerating one fit — different from the score net (§6, maximize-FLOPs per fit) and the SMC gather (§1d, minimize-movement). Roofline-wise it's a throughput/batching problem.
- **SGA linkage is unaffected:** the verdict (`CITestResult` → `EmpiricalCIResult`) is the only cross-boundary object; torch never enters SGA regardless of backend.
